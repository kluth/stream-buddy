## 1. Feature Overview

### Purpose
Configure the Stream Buddy Angular application to run over HTTPS during local development, enabling browser media APIs (`getUserMedia`, `getDisplayMedia`) that require secure contexts. This setup uses locally-trusted SSL certificates generated by `mkcert` to eliminate browser security warnings while maintaining a production-like development environment.

### User-Facing Value
- **Media API Access**: Enables camera, microphone, and screen sharing in local development (these APIs require HTTPS)
- **Developer Experience**: No browser security warnings with locally-trusted certificates
- **Production Parity**: Development environment mirrors production HTTPS behavior
- **Cross-Device Testing**: Test on mobile devices via local network HTTPS
- **Third-Party Integration**: Test OAuth flows and platform integrations that require HTTPS callbacks

### Key Functional Requirements
1. Generate locally-trusted SSL certificates for `localhost` and `127.0.0.1`
2. Configure Angular CLI dev server to serve over HTTPS
3. Install root Certificate Authority (CA) in system trust store
4. Support additional hostnames for local network testing (e.g., `192.168.x.x`)
5. Provide clear setup instructions for all developers
6. Exclude certificates from version control (security)
7. Document certificate renewal process
8. Support multiple development environments (macOS, Windows, Linux)
9. Configure environment detection (dev vs production)
10. Ensure CI/CD pipelines do not require HTTPS certificates

## 2. Research Summary

### Why HTTPS is Required for getUserMedia

**Browser Security Policy:**
- As of Chrome 47 (2015) and Firefox 68 (2019), `getUserMedia` and `getDisplayMedia` APIs only work in **secure contexts**
- A secure context is defined as:
  - HTTPS origin (e.g., `https://example.com`)
  - `localhost` (exempt from HTTPS requirement)
  - `127.0.0.1` (exempt from HTTPS requirement)
  - `file://` protocol (limited use)

**Stream Buddy Implication:**
- Running `ng serve` on `http://localhost:4200` works for `getUserMedia`
- However, testing on mobile devices via local network IP (e.g., `http://192.168.1.100:4200`) FAILS because IP addresses are NOT exempt
- Solution: Serve over HTTPS with valid certificate

**W3C Specification Reference:**
> "The getUserMedia() method MUST only be exposed in secure contexts."
> — Media Capture and Streams Specification (W3C Recommendation)

### mkcert: Best Tool for Local HTTPS (2025)

**What is mkcert:**
- Simple command-line tool to generate locally-trusted SSL certificates
- Automatically creates and installs a local Certificate Authority (CA)
- Supported on macOS, Linux, Windows
- Maintained by Filippo Valsorda (former Go security team member)
- GitHub: https://github.com/FiloSottile/mkcert

**Advantages over Alternatives:**

| Tool | Trust | Setup Complexity | Cross-Platform | Recommendation |
|------|-------|------------------|----------------|----------------|
| mkcert | ✅ Auto-trusted | ⭐ Simple | ✅ Yes | **Recommended** |
| OpenSSL self-signed | ❌ Manual trust | ⭐⭐⭐ Complex | ✅ Yes | Use only if mkcert unavailable |
| Angular CLI --ssl | ❌ Not trusted | ⭐ Simple | ✅ Yes | Generates untrusted certs |
| Let's Encrypt | ✅ Trusted | ⭐⭐⭐⭐ Very complex | ✅ Yes | For production only |

**Why mkcert is Best:**
1. **Zero Configuration**: Installs CA automatically
2. **No Browser Warnings**: Browsers trust certificates immediately
3. **Simple Commands**: `mkcert -install` + `mkcert localhost` = done
4. **Development-Only**: CA only trusted locally, not on other machines
5. **ECDSA Support**: Generates modern elliptic curve certificates (smaller, faster)

### Angular CLI HTTPS Configuration (2025)

**Angular CLI Dev Server (version 20.3.10):**
- Built-in HTTPS support via `ng serve --ssl`
- Configuration options:
  - `--ssl=true`: Enable HTTPS
  - `--ssl-cert=<path>`: Path to SSL certificate file
  - `--ssl-key=<path>`: Path to SSL private key file
  - `--host=<hostname>`: Bind to specific hostname (for local network access)

**angular.json Configuration:**
```json
{
  "projects": {
    "stream-buddy": {
      "architect": {
        "serve": {
          "options": {
            "ssl": true,
            "sslCert": "./tls/localhost-cert.pem",
            "sslKey": "./tls/localhost-key.pem",
            "host": "0.0.0.0"
          }
        }
      }
    }
  }
}
```

**Best Practice:**
- Store HTTPS config in `angular.json` under custom configuration (e.g., `"https"`)
- Default `ng serve` uses HTTP (faster for non-media work)
- Use `ng serve --configuration=https` when testing media features

### Security Best Practices

**Certificate Storage:**
- ✅ Store certificates in `/tls` directory (gitignored)
- ❌ NEVER commit private keys to version control
- ✅ Add `tls/` to `.gitignore`
- ✅ Provide setup script to generate certificates on first run

**Certificate Scope:**
- Generate certificates for:
  - `localhost` (required for Angular dev server)
  - `127.0.0.1` (IPv4 loopback)
  - `::1` (IPv6 loopback)
  - Local network IP (optional, for mobile testing)

**Certificate Lifetime:**
- mkcert default: 10 years
- No need for renewal during development
- If certificate expires, simply regenerate with same command

**Production Considerations:**
- Local CA certificates are ONLY trusted on developer's machine
- Production uses real SSL certificates (Let's Encrypt, AWS Certificate Manager, etc.)
- Ensure CI/CD does NOT require local HTTPS setup

### Cross-Platform Setup Requirements

**macOS:**
- Install mkcert via Homebrew: `brew install mkcert`
- Auto-installs CA in macOS Keychain
- Safari, Chrome, Firefox trust automatically

**Linux:**
- Install mkcert via package manager or GitHub release
- Auto-installs CA in NSS database (Firefox)
- May require manual Chrome/Chromium trust (`certutil`)

**Windows:**
- Install mkcert via Chocolatey: `choco install mkcert`
- Or download from GitHub releases
- Auto-installs CA in Windows Certificate Store
- Edge, Chrome, Firefox trust automatically

**Mobile Device Testing:**
- Install root CA on mobile device manually
- iOS: AirDrop CA file → Settings → Install Profile
- Android: Settings → Security → Install from storage

## 4. Architecture Decisions

### Decision 1: mkcert vs OpenSSL

**Options Considered:**
1. **mkcert**: Automated local CA generation and trust
2. **OpenSSL**: Manual certificate generation with self-signing
3. **Angular CLI default**: Use `ng serve --ssl` without custom certs

**Decision:** Use **mkcert** for local development

**Rationale:**
- **Developer Experience**: mkcert is significantly easier than OpenSSL
- **No Browser Warnings**: Automatically trusted, unlike self-signed OpenSSL certs
- **Cross-Platform**: Works identically on macOS, Linux, Windows
- **Security**: Creates isolated local CA, not system-wide security risk
- **Industry Standard**: Widely adopted in modern web development

**Trade-offs:**
- Requires installing external tool (one-time setup)
- OpenSSL is more "universal" but far more complex
- Accepted trade-off for dramatically better DX

### Decision 2: Configuration Location

**Options Considered:**
1. **angular.json**: Store HTTPS config in Angular CLI config
2. **.angular-cli.json**: Legacy Angular CLI config (deprecated)
3. **Environment Variables**: Pass SSL paths via env vars
4. **npm scripts only**: Configure via command-line flags

**Decision:** Use **angular.json with custom configuration**

**Rationale:**
- **Standard Practice**: Angular CLI config is the idiomatic location
- **Reusability**: Config can be referenced by name (`--configuration=https`)
- **Version Controlled**: Team shares same configuration structure
- **Flexibility**: Can have multiple configs (https, https-mobile, etc.)

**Implementation:**
```json
{
  "configurations": {
    "https": {
      "ssl": true,
      "sslCert": "./tls/localhost-cert.pem",
      "sslKey": "./tls/localhost-key.pem",
      "host": "0.0.0.0"
    }
  }
}
```

**Usage:**
```bash
ng serve --configuration=https
```

### Decision 3: Automated Setup Script

**Options Considered:**
1. **Manual Instructions**: Document steps in README
2. **Shell Script**: Automate with bash script
3. **npm Script**: Use npm postinstall hook
4. **Angular Schematic**: Custom schematic to generate setup

**Decision:** Provide **setup scripts** (bash + PowerShell), NO automatic postinstall

**Rationale:**
- **Explicit Control**: Developers choose when to run setup
- **CI/CD Safety**: Avoids running in CI where mkcert is not installed
- **Cross-Platform**: Separate scripts for Unix and Windows
- **Transparency**: Developers see exactly what's being configured

**Implementation:**
- `/scripts/setup-https.sh` for macOS/Linux
- `/scripts/setup-https.ps1` for Windows
- Documented in README with simple `npm run setup:https` command

**Trade-offs:**
- Requires manual step vs automatic postinstall
- Better control and safety outweigh automation convenience

### Decision 4: Certificate Hostnames

**Options Considered:**
1. **localhost only**: Minimal certificate
2. **localhost + 127.0.0.1 + ::1**: Standard loopback addresses
3. **localhost + IP addresses + custom domains**: Full flexibility

**Decision:** Generate certificate for **localhost, 127.0.0.1, ::1** (standard loopback)

**Rationale:**
- **Standard Development**: Covers 99% of use cases
- **Mobile Testing**: Developers can generate additional cert with their local IP when needed
- **Security**: Smaller certificate scope = better security
- **Documentation**: Provide instructions for adding custom hostnames

**Implementation:**
```bash
mkcert -cert-file ./tls/localhost-cert.pem \
       -key-file ./tls/localhost-key.pem \
       -ecdsa \
       localhost 127.0.0.1 ::1
```

**For Mobile Testing (optional):**
```bash
mkcert -cert-file ./tls/local-network-cert.pem \
       -key-file ./tls/local-network-key.pem \
       -ecdsa \
       localhost 192.168.1.100
```

### 6.1 Setup Script (Unix/macOS/Linux)

**File:** `/scripts/setup-https.sh`

```bash
#!/bin/bash

#
# HTTPS Development Setup Script
# Stream Buddy - Angular Application
#
# This script:
# 1. Checks for mkcert installation
# 2. Installs local CA (if not already installed)
# 3. Generates SSL certificate for localhost
# 4. Creates tls/ directory structure
#

set -e  # Exit on error

# Colors for output
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m' # No Color

# Configuration
TLS_DIR="./tls"
CERT_FILE="${TLS_DIR}/localhost-cert.pem"
KEY_FILE="${TLS_DIR}/localhost-key.pem"

echo "========================================="
echo "  Stream Buddy HTTPS Setup"
echo "========================================="
echo ""

# Step 1: Check for mkcert
echo "Checking for mkcert installation..."
if ! command -v mkcert &> /dev/null; then
    echo -e "${RED}ERROR: mkcert is not installed.${NC}"
    echo ""
    echo "Install mkcert:"
    echo ""
    if [[ "$OSTYPE" == "darwin"* ]]; then
        echo "  macOS:  brew install mkcert"
    elif [[ "$OSTYPE" == "linux-gnu"* ]]; then
        echo "  Linux:  See https://github.com/FiloSottile/mkcert#installation"
    fi
    echo ""
    exit 1
fi

echo -e "${GREEN}✓ mkcert is installed${NC}"
echo ""

# Step 2: Install local CA
echo "Installing local Certificate Authority..."
if mkcert -install; then
    echo -e "${GREEN}✓ Local CA installed successfully${NC}"
else
    echo -e "${YELLOW}⚠ CA may already be installed (this is OK)${NC}"
fi
echo ""

# Step 3: Create tls directory
echo "Creating tls/ directory..."
mkdir -p "${TLS_DIR}"
echo -e "${GREEN}✓ Directory created${NC}"
echo ""

# Step 4: Generate certificate
echo "Generating SSL certificate for localhost..."
if mkcert -cert-file "${CERT_FILE}" \
          -key-file "${KEY_FILE}" \
          -ecdsa \
          localhost 127.0.0.1 ::1; then
    echo -e "${GREEN}✓ Certificate generated successfully${NC}"
else
    echo -e "${RED}ERROR: Failed to generate certificate${NC}"
    exit 1
fi
echo ""

# Step 5: Verify files
echo "Verifying generated files..."
if [[ -f "${CERT_FILE}" ]] && [[ -f "${KEY_FILE}" ]]; then
    echo -e "${GREEN}✓ Certificate files created:${NC}"
    echo "  - ${CERT_FILE}"
    echo "  - ${KEY_FILE}"
else
    echo -e "${RED}ERROR: Certificate files not found${NC}"
    exit 1
fi
echo ""

# Step 6: Success message
echo "========================================="
echo -e "${GREEN}✓ HTTPS setup complete!${NC}"
echo "========================================="
echo ""
echo "You can now run the dev server with HTTPS:"
echo ""
echo -e "  ${YELLOW}npm run start:https${NC}"
echo "  or"
echo -e "  ${YELLOW}ng serve --configuration=https${NC}"
echo ""
echo "Your app will be available at:"
echo -e "  ${GREEN}https://localhost:4200${NC}"
echo ""
echo "Note: Certificate files are in tls/ and are gitignored."
echo ""
```

### 6.6 Environment Detector Service

**File:** `/src/app/core/services/environment-detector.service.ts`

```typescript
import { Injectable } from '@angular/core';

/**
 * Environment configuration
 */
export interface EnvironmentConfig {
  production: boolean;
  isSecureContext: boolean;
  protocol: 'http' | 'https';
  host: string;
  origin: string;
}

/**
 * Security context check result
 */
export interface SecurityContextCheck {
  isSecure: boolean;
  error: string | null;
  suggestion: string | null;
}

/**
 * Service to detect runtime environment and security context
 *
 * @example
 * ```typescript
 * const detector = inject(EnvironmentDetectorService);
 * const check = detector.checkSecurityContext();
 *
 * if (!check.isSecure) {
 *   console.error(check.error);
 *   console.log(check.suggestion);
 * }
 * ```
 */
@Injectable({
  providedIn: 'root'
})
export class EnvironmentDetectorService {
  /**
   * Detect current environment configuration
   */
  detect(): EnvironmentConfig {
    const protocol = window.location.protocol.replace(':', '') as 'http' | 'https';

    return {
      production: this.isProduction(),
      isSecureContext: window.isSecureContext,
      protocol,
      host: window.location.hostname,
      origin: window.location.origin
    };
  }

  /**
   * Check if media APIs (getUserMedia, getDisplayMedia) are available
   * These APIs require a secure context (HTTPS or localhost)
   */
  canUseMediaAPIs(): boolean {
    return window.isSecureContext &&
           'mediaDevices' in navigator &&
           'getUserMedia' in navigator.mediaDevices;
  }

  /**
   * Check security context and provide helpful error messages
   */
  checkSecurityContext(): SecurityContextCheck {
    const env = this.detect();

    // Secure context - all good
    if (env.isSecureContext) {
      return {
        isSecure: true,
        error: null,
        suggestion: null
      };
    }

    // Not secure - provide helpful guidance
    let error: string;
    let suggestion: string;

    if (env.protocol === 'http' && env.host === 'localhost') {
      // This should not happen (localhost is exempt)
      error = 'Secure context not detected on localhost';
      suggestion = 'This is unusual. Try restarting the browser.';
    } else if (env.protocol === 'http') {
      // HTTP on non-localhost
      error = 'Media APIs require HTTPS or localhost';
      suggestion = env.production
        ? 'Ensure your production server is configured with HTTPS.'
        : 'Run the dev server with: npm run start:https';
    } else {
      // Unknown issue
      error = 'Secure context not available';
      suggestion = 'Ensure you are using HTTPS or localhost.';
    }

    return {
      isSecure: false,
      error,
      suggestion
    };
  }

  /**
   * Get error message if not in secure context (null if secure)
   */
  getSecureContextError(): string | null {
    const check = this.checkSecurityContext();
    return check.error;
  }

  /**
   * Detect if running in production mode
   */
  private isProduction(): boolean {
    // In production builds, Angular removes this check
    // Use environment file for actual production detection
    return false; // Development default
  }
}
```

# HTTPS Development Setup Guide

## Why HTTPS is Required

Stream Buddy uses browser media APIs (`getUserMedia`, `getDisplayMedia`) to access your camera, microphone, and screen. Modern browsers **require HTTPS** for these APIs as a security measure.

### Secure Contexts

Browsers allow media access only in "secure contexts":
- ✅ `https://` URLs (with valid SSL certificate)
- ✅ `localhost` (exempt from HTTPS requirement)
- ✅ `127.0.0.1` (exempt from HTTPS requirement)
- ❌ `http://192.168.x.x` (local network IP over HTTP - **NOT allowed**)

**Problem:** Testing on mobile devices via local network IP requires HTTPS.

**Solution:** Use `mkcert` to generate locally-trusted SSL certificates.

---

## Quick Start

### Prerequisites
- Node.js 18+ and npm installed
- Stream Buddy repository cloned
- Terminal/Command Prompt access

### One-Command Setup

**macOS / Linux:**
```bash
npm run setup:https
```

**Windows:**
```powershell
npm run setup:https
```

This script will:
1. Check if mkcert is installed (prompt to install if missing)
2. Install a local Certificate Authority (CA) on your machine
3. Generate SSL certificate for localhost
4. Create `tls/` directory with certificate files

### Start HTTPS Dev Server

```bash
npm run start:https
```

Your app will be available at: **https://localhost:4200**

brew install mkcert
```

       localhost 127.0.0.1 ::1
```

**What this does:**
- Creates `tls/` directory
- Generates ECDSA certificate (modern, smaller than RSA)
- Covers `localhost`, `127.0.0.1`, and `::1` (IPv6 loopback)

### Step 4: Verify Setup

Check that files exist:
```bash
ls -la tls/
```

You should see:
```
tls/
├── localhost-cert.pem
└── localhost-key.pem
```

### Step 5: Start Dev Server

```bash
npm run start:https
```

Visit **https://localhost:4200** - you should see a **secure** connection (green padlock in browser).

To test on a mobile device via your local network IP:

### Step 1: Find Your Local IP Address

**macOS/Linux:**
```bash
ifconfig | grep "inet "
```

**Windows:**
```powershell

```

Look for an address like `192.168.1.100`.

### Step 2: Generate Certificate with Your IP

```bash
mkcert -cert-file ./tls/local-network-cert.pem \
       -key-file ./tls/local-network-key.pem \
       -ecdsa \
       localhost 192.168.1.100
```

Replace `192.168.1.100` with your actual IP.

### Step 3: Update angular.json

Edit `angular.json` to use the new certificate:

```json
{
  "configurations": {
    "https": {
      "sslCert": "./tls/local-network-cert.pem",
      "sslKey": "./tls/local-network-key.pem",
      "host": "0.0.0.0"
    }
  }
}
```

**iOS:**
1. Find the CA file: `$(mkcert -CAROOT)/rootCA.pem`
2. AirDrop it to your iPhone/iPad

4. Go to Settings → General → About → Certificate Trust Settings
5. Enable full trust for the mkcert CA

**Android:**
1. Find the CA file: `$(mkcert -CAROOT)/rootCA.pem`
2. Transfer to device (email, USB, etc.)
3. Go to Settings → Security → Install from storage
4. Select the CA file
5. Name it "mkcert Development CA"

Start the server:
```bash
npm run start:https
```

On your mobile device, visit:
```
https://192.168.1.100:4200
```

(Replace with your actual IP)

## Troubleshooting

### "mkcert: command not found"

**Solution:** Install mkcert (see Step 1 above).

### Browser Shows "Not Secure" Warning

**Cause:** Local CA not installed or not trusted.

**Solution:**
1. Run `mkcert -install` again
2. Restart your browser
3. Check that CA is in system trust store:
   - macOS: Open Keychain Access → System → Find "mkcert"
   - Windows: Run `certmgr.msc` → Trusted Root CAs → Find "mkcert"
   - Linux: Check `~/.local/share/mkcert/rootCA.pem` exists

### Certificate Expired

**Cause:** mkcert certificates last 10 years, but system clock may be wrong.

**Solution:**
1. Check system date/time
2. Regenerate certificate: `mkcert localhost 127.0.0.1 ::1`

### "Address already in use" Error

**Cause:** Port 4200 is already in use.

**Solution:**
```bash
# Find process using port 4200
lsof -i :4200   # macOS/Linux
netstat -ano | findstr :4200   # Windows

# Kill the process or use a different port
ng serve --configuration=https --port=4201
```

### Mobile Device Shows "Cannot Connect"

**Checklist:**
- [ ] Firewall allows port 4200
- [ ] Mobile device on same Wi-Fi network
- [ ] Used correct local IP address
- [ ] Certificate includes the IP address (Step 2 above)
- [ ] CA installed on mobile device (Step 4 above)

## 8. Security Considerations

### Certificate Security

**Private Key Protection:**
- Private keys stored in `tls/` directory (gitignored)
- Never commit `.pem`, `.key`, `.crt` files to version control
- If accidentally committed, rotate certificate immediately

**CA Security:**
- mkcert CA (`rootCA-key.pem`) stored in `~/.local/share/mkcert/` (user home)
- CA private key grants ability to sign certificates
- **Never share CA private key** - it's local to your machine only

**Certificate Scope:**
- Certificates only trusted on machines with CA installed
- Certificates only valid for specified hostnames (localhost, IPs)
- No wildcard certificates generated

### Production vs Development

**Development (mkcert):**
- Locally-trusted certificates
- Not trusted by other machines
- Perfect for local development

**Production (Real SSL):**
- Use Let's Encrypt, AWS Certificate Manager, Cloudflare, etc.
- Globally trusted certificates
- Automatic renewal

**Separation:**
- Environment detection in code (`window.isSecureContext`)
- Never reference local certificates in production builds
- Angular environments handle dev vs prod URLs

## 9. Performance Considerations

### Development Server Performance

**HTTPS Overhead:**
- Minimal impact on development server performance
- TLS handshake adds ~50ms on initial connection
- Subsequent requests use TLS session resumption (fast)

**Certificate Type:**
- ECDSA certificates are smaller and faster than RSA
- Modern browsers prefer ECDSA
- mkcert uses ECDSA by default with `-ecdsa` flag

**Benchmark:**
- HTTP: ~10ms initial connection
- HTTPS (ECDSA): ~60ms initial connection
- HTTPS (RSA): ~100ms initial connection

**Impact:** Negligible for development (one-time cost per page load)

## 12. Appendix

### Platform-Specific Notes

**macOS:**
- mkcert installs CA in Keychain Access → System
- Safari, Chrome, Edge trust automatically
- Firefox uses its own certificate store (auto-handled by mkcert)

**Linux:**
- mkcert installs CA in NSS database (`~/.pki/nssdb`)
- Chrome/Chromium may require `libnss3-tools` package
- Firefox auto-trusts if NSS tools installed

**Windows:**
- mkcert installs CA in Windows Certificate Store
- All browsers (Chrome, Edge, Firefox) trust automatically
- May require admin privileges for CA installation

### Certificate Details

**Default mkcert Certificate:**
- **Algorithm:** ECDSA P-256 (with `-ecdsa` flag)
- **Validity:** 10 years
- **Subject:** `mkcert <username>@<hostname>`
- **SANs:** All hostnames specified in command

**Example Certificate Info:**
```
Subject: CN=mkcert developer@macbook
Issuer: CN=mkcert developer@macbook
Not Before: Nov 15 12:00:00 2025 GMT
Not After:  Nov 15 12:00:00 2035 GMT
Subject Alternative Names:
  DNS:localhost
  IP:127.0.0.1
  IP:::1
```

### Useful Commands

**Check certificate validity:**
```bash
openssl x509 -in tls/localhost-cert.pem -text -noout
```

**Find mkcert CA location:**
```bash
mkcert -CAROOT
```

**List all trusted certificates (macOS):**
```bash
security find-certificate -a -p /Library/Keychains/System.keychain
```

**List all trusted certificates (Linux):**
```bash
certutil -L -d sql:$HOME/.pki/nssdb
```

**Test HTTPS connection:**
```bash
curl -v https://localhost:4200
```
